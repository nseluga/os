## What the project actually does

A two-person Harvey Mudd CS158 course project ("Context-Aware Shot Value Estimation in the NBA Using Machine Learning," Nate Seluga & Michael O'Brien — see `Project_Writeup.pdf`, 6 pp.). It is a **binary make/miss classification** problem on the classic **2014–15 NBA shot-logs dataset** (`data/raw/shot_logs.csv`, 128,069 shots, 21 columns; matchup dates like "MAR 04, 2015"). The dataset has **no x/y court coordinates** — only shot distance, closest-defender distance, shot clock, dribbles, touch time, period, home/away. Notebook 08's own header states: "The processed dataset does not include x/y court coordinates, so location is represented by shot zone and shot-distance bins."

Pipeline (9 notebooks):
1. `01_data_import_cleaning.ipynb` — cleaning, invalid-value filtering, mislabeled 2s/3s inspection (774 sub-22-ft "threes," 385 24+-ft "twos"); modeling notebooks use 122,203 rows after dropping missing shot clock.
2. `02_feature_enginereing.ipynb` — 15 engineered features (shot-type flags, `defender_distance_ratio`, `time_pressure_index`, `def_dist_x_shot_dist`, `shot_dist_squared`, `creation_load_log`, etc.).
3. `03_eda.ipynb` — distributions, FG% by distance/defender bins, correlation heatmap (overall FG% = 0.452).
4–7. Logistic regression and Decision Tree / Random Forest / XGBoost, each run raw vs. engineered; nb07 adds RandomizedSearchCV with stratified 3-fold CV (tuned on the training split only), saves `models/xgboost_engineered_shot_model.joblib` (22 features).
8. `08_shot_data_visualizations.ipynb` — defines **expected shot value = predicted make probability × pts_type**; Plotly EV-vs-distance-by-pressure chart, value-vs-frequency bubble chart, and an ipywidgets **interactive zone × distance-bin heatmap** (not a 500-cell court grid).
9. `09_shot_uncertainty_and_shap.ipynb` — calibration/uncertainty scatter (6,000 test shots) and full SHAP suite (beeswarm, 6-panel dependence grid, 22×22 interaction heatmap).

Data files are not committed (empty `.gitkeep` in `data/raw` and `data/processed`).

## Real findings/results (exact numbers from repo outputs)

Test set = random 80/20 split of one season (97,762 train / 24,441 test). Not a season holdout.

Live notebook cell outputs:
- LR baseline (nb04, 8 features): Accuracy 0.6094, ROC AUC 0.6279, Log Loss 0.6566 (naive baseline log loss 0.6895).
- LR engineered (nb06, 23 features): Accuracy 0.6122, AUC 0.6310, Log Loss 0.6552, Brier 0.2319 — engineered features add ~0.003 AUC over baseline for LR.
- Trees raw (nb05): DT AUC 0.6276; RF AUC 0.6355; XGBoost Accuracy 0.6195, AUC 0.6360, Log Loss 0.6496.
- Trees tuned + engineered (nb07): DT AUC 0.6283; RF AUC 0.6346; XGBoost Accuracy 0.6199, AUC 0.6365, Log Loss 0.6494, Brier 0.2296. Best CV AUCs: DT 0.6318, RF 0.6371, XGB 0.6384.
- PDF Table 1 (canonical results): raw features — LR 0.6300 / DT 0.6293 / RF 0.6397 / XGB 0.6411 AUC; engineered — LR 0.6302 / DT 0.6283 / RF 0.6346 / XGB 0.6365 AUC.

Honest headline conclusion (stated repeatedly in nb05/06/07 and the PDF): **performance plateaus at AUC ~0.63–0.64 regardless of model class or feature engineering**; the missing signal is shooter skill, which is deliberately excluded (player-agnostic model). Tuned engineered trees actually came in slightly *below* the nb05 raw baselines (XGB 0.6365 vs 0.6416, delta −0.0051 per nb07's own summary table). Calibration curves show all models well-calibrated; SHAP ranks `defender_distance_ratio` as the top feature, ahead of `shot_dist` and `touch_time`.

EV findings from `notebooks/value_defender.png` / nb08: open (6+ ft) shots at 0–4 ft ≈ 1.85 EV; very tight 20–22 ft shots ≈ 0.62 EV; a clear EV bump at the 22–24 ft "corner 3" bin (≈0.87 very tight to ≈1.20 open); mid-range (8–22 ft) is a trough across all pressure levels.

Internal inconsistencies an auditor should know about:
- nb05's summary text cites XGB AUC 0.642 / RF 0.640, but the live cell outputs show 0.6360 / 0.6355 — stale numbers from an earlier run; nb07's hardcoded "nb05 baseline" rows (0.6397/0.6416) likewise don't match nb05's current outputs. The PDF uses the stale (higher) baselines.
- nb03 and nb09 load a 127,757-row version of the cleaned CSV (with a `shot_clock_missing` flag) while nb01's committed code produces 122,203 rows; nb09's 80/20 split (102,205/25,552) therefore differs from the split the model was trained on, so its "test set" SHAP/calibration sample can overlap training rows.

## Inaccuracies & gaps in current writeup (writeup claim → repo reality)

1. "2.1M shots (2019–2024)" (frontmatter + Data section) → **128,069 shots from the 2014–15 season only** (122,203 after cleaning). Both volume (~16× inflated) and seasons are wrong.
2. "Source: NBA Stats API (stats.nba.com)" → the raw file is the well-known Kaggle 2014–15 shot-logs CSV; `nba_api` is in requirements but no notebook calls it.
3. "Model Accuracy: R² = 0.68" → **no R² exists anywhere in the repo**; this is a classification project. Real numbers: accuracy ~0.62, ROC AUC 0.6365 (tuned engineered XGBoost) / 0.6411 (PDF raw baseline). An R² framing also misrepresents the task.
4. "MAE ±0.15 expected points per shot" → never computed. Real calibration metrics: log loss 0.6494, Brier 0.2296.
5. "Precision by location type: 0.72 / 0.65 / 0.58" → no per-zone precision analysis exists anywhere.
6. "500-cell hexagonal court grid," "Locations Gridded: 500-cell court" → no court gridding at all; **the dataset has no x/y coordinates** (nb08 says so explicitly). Location = 3 zones (Paint / Mid-range / 3PT) × 11 distance bins.
7. "Location (x, y): absolute court coordinates" and "Angle: 0–360°, bucketed into zones" → neither feature exists. `location` in the data is home/away.
8. "Interpolation: smoothed sparse cells using nearest-neighbor averaging" → not done; sparse bins are simply masked below a min-attempts threshold (75).
9. "Validation: Holdout 2024 season" → random 80/20 train_test_split of a single 2014–15 season; no temporal holdout.
10. "7 variables" context features → 6 primary raw features plus 15 engineered ones; final model uses 22 features.
11. "Defender proximity (0–10 feet)" → actual `close_def_dist` range 0–53.2 ft.
12. "Shooter context: player experience (rookie, star, veteran)" and "Stars shoot 2–4% better... Superstars 1.2 EV vs role players 1.15 EV" → **no player-skill feature of any kind was modeled**; the project is explicitly player-agnostic, and every notebook names shooter skill as the *unmeasured* variable causing the AUC ceiling. These numbers are invented.
13. The ECharts "Expected Value Heatmap" data points ([0,10,1.3], [23.75,14,0.95], etc.) → hand-typed values with no corresponding repo output (nothing in `/home/user/portfolio/src/data/` either — only `pitcher-injury-risk.json` and `site.ts`). The repo cannot produce an x/y court map. Real EV numbers that could replace them exist in nb08/value_defender.png (e.g. open 0–4 ft ≈ 1.85; very tight 20–22 ft ≈ 0.62).
14. "Corner threes slightly worse than wing threes (geometry—longer distance)" → backwards even internally (the writeup later says corners are *better*), and the repo has no angle data; it identifies corner 3s only as the 22–24 ft distance bin, where EV is indeed elevated.
15. "R² ... explains 68% of shot-outcome variance" framing hides the project's actual, more interesting result: shot outcomes are barely predictable (AUC ~0.64) and the models are valuable because they are *well-calibrated*, not discriminative.
16. Frontmatter date 2024-05-10 → repo commit is 2026-07-01 and PDF/course context suggest a 2025–26 course project (data itself is 2014–15).

**Real strengths the writeup never mentions:**
- Full SHAP interpretability suite (beeswarm, dependence grid, interaction matrix) — the most impressive technical artifact in the repo.
- Proper hyperparameter tuning hygiene: RandomizedSearchCV, stratified 3-fold CV on the training split only, test set touched once.
- Calibration analysis (curves, Brier score) and the standout uncertainty visualization ("wide-open shots miss, impossible shots go in").
- Honest negative results: engineered features add ~0.003 AUC for LR and nothing for trees; tuned models slightly underperform earlier baselines; explicit "predictability ceiling" argument attributing residual variance to shooter skill.
- Four-model comparison (LR, DT, RF, XGB) run twice (raw vs engineered) with a clean results table (PDF Table 1).
- Basketball-literate feature engineering (catch-and-shoot, pull-up, iso, creation load, defender-distance ratio) and data-quality forensics (mislabeled 2s/3s).
- Interactive Plotly/ipywidgets dashboards; a formal NeurIPS-style PDF writeup with literature citations (Goldsberry CourtVision; Shah & Romijnders).
- Collaborative project with documented contribution split.

## Recommended framing for rewrite

Frame it as an honest calibration-first study: "We built player-agnostic make-probability models (logistic regression → tuned XGBoost) on 122K shots from the 2014–15 NBA tracking shot logs and found a hard predictability ceiling — AUC ~0.64 no matter the model or feature set — but the probabilities are well-calibrated, so multiplying them by shot value yields trustworthy expected-points-per-shot estimates." Lead with the sport insight the repo actually produces (pressure-stratified EV curves: corner-3 bump, mid-range trough, defender proximity via `defender_distance_ratio` as the top SHAP feature) and with the uncertainty/SHAP visuals, which are the project's genuine differentiators. Kill every fabricated number (2.1M shots, R² 0.68, 500-cell grid, MAE, per-zone precision, star-vs-role-player effects) and replace the invented court heatmap with the real charts: value_defender, shot_uncertainty_calibration, and the SHAP beeswarm.
