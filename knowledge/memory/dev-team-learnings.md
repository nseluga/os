Cross-project lessons from dev-team runs. Project-specific findings stay in each repo's `.claude/dev-team/team-memory.md`.

- 2026-07-09 When a change adds `<details>/<summary>` elements inside repeating components (cards), pre-existing tests that use first-match regex for `<details>` will silently match the wrong element — **Why:** regex position assumptions break when more elements of the same type are added. **How to apply:** fixture tests involving HTML elements that may appear multiple times (details, button, summary) should match by content (e.g. "Completed" text, aria-label) rather than by DOM position.

- 2026-07-09 `readManual()` must use explicit per-field construction — never `{ ...EMPTY_MANUAL, ...parsed }` — **Why:** shallow spread returns references to the constant's nested objects; callers that mutate those objects (e.g. `manual.hidden_fields[id][field] = true`) poison `EMPTY_MANUAL` for all subsequent calls in the same process, causing hard-to-reproduce state bleed. **How to apply:** always construct `{ field1: {...(parsed.field1 ?? {})}, field2: [...(parsed.field2 ?? [])], ... }` explicitly. Any new field added to the data model must be added to this construction simultaneously.

- 2026-07-09 Expensive I/O (git shell calls via getMergedProjects/getProjects, disk reads) must never run inside a write mutex — **Why:** the lock serializes all concurrent requests for the full duration of the slowest call; a git subprocess can take seconds. **How to apply:** resolve any external data (project list, git log) before `runExclusive`; the mutex should protect only the read-modify-write step on the shared file.

- 2026-07-09 Before a color palette sweep, audit test files for color-anchored assertions (`/text-gray-/`, `/text-blue-/`, etc.) and update them in the same commit as the sweep — **Why:** the sweep will break those assertions, and the fix-as-follow-up burns an extra QA attempt. **How to apply:** run `grep -r 'text-gray-\|text-blue-' tests/` before executing the sweep; patch the assertions alongside the component changes.

- 2026-07-09 `focus-visible:ring-*` on filled colored buttons must use a white ring with offset, not a lighter tint of the button color — **Why:** `ring-indigo-400` on `bg-indigo-600` achieves ~2.4:1 contrast, below the WCAG 3:1 minimum for non-text UI components. **How to apply:** use `focus-visible:ring-white focus-visible:ring-offset-2 focus-visible:ring-offset-[button-color]` on any filled-color button.

- 2026-07-09 Keyword-matching functions (autoTag, search, filter) must use word-boundary checks, not bare `indexOf` — **Why:** substring matches cause false positives ("os" matches inside "gross", "ml" inside "email", "ai" inside "said"). **How to apply:** implement `isWordBoundary(text, start, end)` checking `\W` or string edge on both sides of the match; never ship a keyword scanner that uses only `indexOf`.

- 2026-07-09 When composed SSR pages need both getMergedProjects() output AND raw manual data (token_log, notes, etc.), use an optional `manual?` parameter on getMergedProjects() to share a single disk read — **Why:** without this, getMergedProjects() calls readManual() internally while the page also calls it directly, doubling disk I/O per render. **How to apply:** `const manual = await readManual(); const projects = await getMergedProjects(manual);` — one read, zero redundancy.

- 2026-07-09 `claude -p --bare` removes the Skill tool from the model's tool list entirely — **Why:** `--bare` strips all non-essential tools including the Skill invocation mechanism; any eval rung that relies on skills silently gets zero skill access, not degraded access. **How to apply:** never use `--bare` for eval rungs that test skill-augmented behavior; use non-bare mode with `CLAUDE_CONFIG_DIR` + `ANTHROPIC_AUTH_TOKEN` + an isolated skill dir instead. Probe the flag empirically before building against it.

- 2026-07-09 Headless one-shot eval tasks must use pure unit tests, not integration tests that require a running dev server — **Why:** a check.sh that needs `npm run dev` to be live cannot pass in a subprocess that exits immediately; the task silently fails on infra, not logic, poisoning the result matrix. **How to apply:** when authoring eval tasks, verify the check runs to completion with the server stopped; if it doesn't, either convert to a pure unit test or mark the task as unsuitable for headless evaluation.

- 2026-07-17 When a correction pass updates source files, downstream summary surfaces (about pages, llms.txt, READMEs, config descriptions) are the most common miss — **Why:** engineers scope their grep/edit to the files that contain the error, not every file that echoes the same value. A correction in `pitcher-injury-risk.mdx` leaves `about.astro` and `llms.txt` stale. **How to apply:** for any pass that corrects factual claims, explicitly instruct the engineer to grep all summary/description surfaces for the old value and update them in the same commit.

- 2026-07-17 Pattern sweeps (em dash, naming, style, banned vocab) must cover ALL authored content files, not just published/active ones — **Why:** draft:true or hidden files still contain authored prose subject to the same voice rules; scoping the sweep to live files causes QA to fail on the broader grep criterion. **How to apply:** run pattern greps over the full content directory (`grep -rn 'pattern' src/content/`), never a filtered subset; the done-when criterion should name the full directory, not a page list.

- 2026-07-19 In a truth-in-marketing / copy-correction item, an engineer reframing claims will tend to assert an unbuilt capability as a present-tense guarantee (e.g. "you can export your data any time" when no export feature exists) — **Why:** the reframe fixes the flagged false claim but introduces a new one because the engineer reasons about desired positioning, not shipped features. **How to apply:** for any content/marketing item, add a review check that every present-tense capability claim maps to a feature that actually exists in the repo; phrase not-yet-built capabilities as intent/service ("we'll export it whenever you ask"), never as a self-serve guarantee. Grep the repo for the claimed capability to confirm it exists before passing the item.

- 2026-07-19 An exported config/constant that serves as a shared source-of-truth must be DEEP-frozen at runtime, not just typed `Readonly<>`/`as const` — **Why:** TypeScript `Readonly` and `as const` are compile-time only; nested object fields stay runtime-mutable, so any consumer can poison the shared module for the whole process (`PRICING.standard.monthlyCents = 1`). Same failure family as the EMPTY_MANUAL shallow-spread bug, in a new form. **How to apply:** for any module-level constant that is imported and read by others (pricing configs, default option objects, enum-like maps), `Object.freeze` the top object AND each nested object, mark fields `readonly`, and add a test asserting a mutation attempt does not change the value. Flag this in review for every new shared package.

- 2026-07-19 dt-* agents write their `*-report.md` to the working tree but often do NOT commit it; the orchestrator must commit reports itself when recording each item's outcome, and must never `git checkout`/discard a report file expecting the latest agent's version to persist — **Why:** in an A2 run the engineer-report was written but not committed with the feature commit, then a `git checkout --` to undo a different agent's stray edit destroyed it, forcing a manual reconstruction; a mid-run agent that dies (API socket error) also leaves reports stale/uncommitted. **How to apply:** treat `.claude/dev-team/*.md` as orchestrator-owned artifacts — `git add .claude/dev-team/ ` in the same commit that records the PROGRESS/PLAN outcome; before discarding any uncommitted working-tree change, confirm it isn't the only copy of a just-written report. Also: any agent that starts a dev server must run it backgrounded with a bounded curl-poll and always kill it (no orphan servers).
