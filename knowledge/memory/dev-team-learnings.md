Cross-project lessons from dev-team runs. Project-specific findings stay in each repo's `.claude/dev-team/team-memory.md`.

- 2026-07-09 When a change adds `<details>/<summary>` elements inside repeating components (cards), pre-existing tests that use first-match regex for `<details>` will silently match the wrong element — **Why:** regex position assumptions break when more elements of the same type are added. **How to apply:** fixture tests involving HTML elements that may appear multiple times (details, button, summary) should match by content (e.g. "Completed" text, aria-label) rather than by DOM position.

- 2026-07-09 `readManual()` must use explicit per-field construction — never `{ ...EMPTY_MANUAL, ...parsed }` — **Why:** shallow spread returns references to the constant's nested objects; callers that mutate those objects (e.g. `manual.hidden_fields[id][field] = true`) poison `EMPTY_MANUAL` for all subsequent calls in the same process, causing hard-to-reproduce state bleed. **How to apply:** always construct `{ field1: {...(parsed.field1 ?? {})}, field2: [...(parsed.field2 ?? [])], ... }` explicitly. Any new field added to the data model must be added to this construction simultaneously.

- 2026-07-09 Expensive I/O (git shell calls via getMergedProjects/getProjects, disk reads) must never run inside a write mutex — **Why:** the lock serializes all concurrent requests for the full duration of the slowest call; a git subprocess can take seconds. **How to apply:** resolve any external data (project list, git log) before `runExclusive`; the mutex should protect only the read-modify-write step on the shared file.

- 2026-07-09 Before a color palette sweep, audit test files for color-anchored assertions (`/text-gray-/`, `/text-blue-/`, etc.) and update them in the same commit as the sweep — **Why:** the sweep will break those assertions, and the fix-as-follow-up burns an extra QA attempt. **How to apply:** run `grep -r 'text-gray-\|text-blue-' tests/` before executing the sweep; patch the assertions alongside the component changes.

- 2026-07-09 `focus-visible:ring-*` on filled colored buttons must use a white ring with offset, not a lighter tint of the button color — **Why:** `ring-indigo-400` on `bg-indigo-600` achieves ~2.4:1 contrast, below the WCAG 3:1 minimum for non-text UI components. **How to apply:** use `focus-visible:ring-white focus-visible:ring-offset-2 focus-visible:ring-offset-[button-color]` on any filled-color button.

- 2026-07-09 Keyword-matching functions (autoTag, search, filter) must use word-boundary checks, not bare `indexOf` — **Why:** substring matches cause false positives ("os" matches inside "gross", "ml" inside "email", "ai" inside "said"). **How to apply:** implement `isWordBoundary(text, start, end)` checking `\W` or string edge on both sides of the match; never ship a keyword scanner that uses only `indexOf`.

- 2026-07-09 When composed SSR pages need both getMergedProjects() output AND raw manual data (token_log, notes, etc.), use an optional `manual?` parameter on getMergedProjects() to share a single disk read — **Why:** without this, getMergedProjects() calls readManual() internally while the page also calls it directly, doubling disk I/O per render. **How to apply:** `const manual = await readManual(); const projects = await getMergedProjects(manual);` — one read, zero redundancy.

- 2026-07-09 `claude -p --bare` removes the Skill tool from the model's tool list entirely — **Why:** `--bare` strips all non-essential tools including the Skill invocation mechanism; any eval rung that relies on skills silently gets zero skill access, not degraded access. **How to apply:** never use `--bare` for eval rungs that test skill-augmented behavior; use non-bare mode with `CLAUDE_CONFIG_DIR` + `ANTHROPIC_AUTH_TOKEN` + an isolated skill dir instead. Probe the flag empirically before building against it.

- 2026-07-09 Headless one-shot eval tasks must use pure unit tests, not integration tests that require a running dev server — **Why:** a check.sh that needs `npm run dev` to be live cannot pass in a subprocess that exits immediately; the task silently fails on infra, not logic, poisoning the result matrix. **How to apply:** when authoring eval tasks, verify the check runs to completion with the server stopped; if it doesn't, either convert to a pure unit test or mark the task as unsuitable for headless evaluation.

- 2026-07-17 When a correction pass updates source files, downstream summary surfaces (about pages, llms.txt, READMEs, config descriptions) are the most common miss — **Why:** engineers scope their grep/edit to the files that contain the error, not every file that echoes the same value. A correction in `pitcher-injury-risk.mdx` leaves `about.astro` and `llms.txt` stale. **How to apply:** for any pass that corrects factual claims, explicitly instruct the engineer to grep all summary/description surfaces for the old value and update them in the same commit.

- 2026-07-17 Pattern sweeps (em dash, naming, style, banned vocab) must cover ALL authored content files, not just published/active ones — **Why:** draft:true or hidden files still contain authored prose subject to the same voice rules; scoping the sweep to live files causes QA to fail on the broader grep criterion. **How to apply:** run pattern greps over the full content directory (`grep -rn 'pattern' src/content/`), never a filtered subset; the done-when criterion should name the full directory, not a page list.

- 2026-07-19 In a truth-in-marketing / copy-correction item, an engineer reframing claims will tend to assert an unbuilt capability as a present-tense guarantee (e.g. "you can export your data any time" when no export feature exists) — **Why:** the reframe fixes the flagged false claim but introduces a new one because the engineer reasons about desired positioning, not shipped features. **How to apply:** for any content/marketing item, add a review check that every present-tense capability claim maps to a feature that actually exists in the repo; phrase not-yet-built capabilities as intent/service ("we'll export it whenever you ask"), never as a self-serve guarantee. Grep the repo for the claimed capability to confirm it exists before passing the item.

- 2026-07-19 An exported config/constant that serves as a shared source-of-truth must be DEEP-frozen at runtime, not just typed `Readonly<>`/`as const` — **Why:** TypeScript `Readonly` and `as const` are compile-time only; nested object fields stay runtime-mutable, so any consumer can poison the shared module for the whole process (`PRICING.standard.monthlyCents = 1`). Same failure family as the EMPTY_MANUAL shallow-spread bug, in a new form. **How to apply:** for any module-level constant that is imported and read by others (pricing configs, default option objects, enum-like maps), `Object.freeze` the top object AND each nested object, mark fields `readonly`, and add a test asserting a mutation attempt does not change the value. Flag this in review for every new shared package.

- 2026-07-19 dt-* agents write their `*-report.md` to the working tree but often do NOT commit it; the orchestrator must commit reports itself when recording each item's outcome, and must never `git checkout`/discard a report file expecting the latest agent's version to persist — **Why:** in an A2 run the engineer-report was written but not committed with the feature commit, then a `git checkout --` to undo a different agent's stray edit destroyed it, forcing a manual reconstruction; a mid-run agent that dies (API socket error) also leaves reports stale/uncommitted. **How to apply:** treat `.claude/dev-team/*.md` as orchestrator-owned artifacts — `git add .claude/dev-team/ ` in the same commit that records the PROGRESS/PLAN outcome; before discarding any uncommitted working-tree change, confirm it isn't the only copy of a just-written report. Also: any agent that starts a dev server must run it backgrounded with a bounded curl-poll and always kill it (no orphan servers).

- 2026-07-20 For in-place migration edits (no data exists yet), keep every DDL change idempotent with guarded `DO $$`/`IF NOT EXISTS`/`DROP ... IF EXISTS` blocks AND wire Critical-invariant regressions (FK on-delete behavior, status-gating triggers) into the EXISTING local-DB test harness rather than a new test file — **Why:** a schema whose "fix" isn't re-apply-safe breaks `supabase db push`/`db:test:local` idempotency on an already-applied DB, and a Critical caught by review (cascade-delete history loss, a declined record becoming a live job) silently regresses later unless an executable invariant guards it. **How to apply:** when a data-path review returns Critical schema findings, have dt-fix edit migrations in place with guarded blocks, then add a harness assertion that the bad operation RAISES (delete-restrict, confirmed-only insert) and a good one succeeds; re-run the harness as the QA re-gate. Prefer FORCE (not ENABLE) RLS and integer-cents money — Opus review reliably catches these on flag:data-path items, so budget the Opus review there.

- 2026-07-21 A DB helper that reads a JWT/session claim and casts it (e.g. `nullif(claim,'')::uuid`) must gate the cast so a malformed (non-empty, wrong-format) value resolves to NULL, never throws — **Why:** RLS/policy helpers run on every query; a raw `::uuid`/`::int` cast on a garbled or tampered claim throws `invalid input syntax`, breaking every query for that user (a self-inflicted DoS) instead of failing safe to zero-rows. Absent/blank claims usually already fail safe; malformed ones are the gap. **How to apply:** regex-gate the value before casting (`case when v ~ '^[0-9a-fA-F]{8}-...$' then v::uuid else null end`) keeping the function pure/STABLE; add a fixture impersonating a user whose claim is `"not-a-uuid"` and assert zero rows + no throw. Absent, blank, AND malformed must all fail closed.
- 2026-07-21 Supabase RLS is testable on a vanilla local Postgres by emulating the auth layer — **Why:** policies reference `auth.uid()`/`auth.jwt()` and roles `anon`/`authenticated`/`service_role` that don't exist in stock PG, so teams wrongly assume RLS can only be tested against a live Supabase. **How to apply:** in the ephemeral test cluster create `schema auth` with `auth.jwt()` = `coalesce(current_setting('request.jwt.claims',true),'{}')::jsonb` (+ `auth.uid()`), create the three roles, and impersonate per-case with `select set_config('request.jwt.claims', <json>, true); set local role authenticated;`. NOTE PG version: `security_invoker` views are PG15+ (Supabase) and error on PG14 — gate those cases on `server_version_num>=150000` and treat base-table RLS + column GRANTs (version-independent) as the real, fully-testable guarantee.
- 2026-07-21 Route-middleware gating gives false authz confidence in Next.js — server actions are POSTable from ANY path, bypassing the matcher. **Why:** engineer shipped 6 unauthenticated mutating actions behind a "protected" /admin matcher; QA's happy-path smoke missed it, only review caught it. **How to apply:** in any Next.js repo, reviewers/QA must verify each mutating server action checks authz itself (shared fail-closed helper as the first statement); never accept middleware as the guard.
- 2026-07-21 Every agent that runs the test suite must be told the exact expected pass/skip counts and how to wire env (e.g. DATABASE_URL from .env.local) in its spawn prompt — not just QA — **Why:** spawned agents get fresh shells; env-gated integration files skip silently, so a builder/UI agent reports "tests pass" while the DB-backed half never ran (recurred twice in one run: fixer item 8, dt-ui item 10). **How to apply:** the orchestrator states "expect N pass / M skip; if skips exceed M, your env is wrong — fix it before committing" in every spawn prompt that includes a test run.
- 2026-07-21 JS money-path guards must use `Number.isSafeInteger`, never bare `Number.isInteger` — and reviewers' prescribed fixes can themselves be wrong on fractional intermediates. **Why:** `Number.isInteger(2**53)` is true, so an "integer cents" assert built on it silently admits precision-lost values; the gap threads through single-value asserts, post-reduce subtotals, AND `total * pct` products (which overflow 2^53 before the /100). A reviewer prescribed `isSafeInteger(product)`, but the product is legitimately fractional (pct 66.67) — the correct guard there is magnitude (`product > MAX_SAFE_INTEGER`). Adversarial "overflow" probes that deliberately stay in safe range never catch any of this. **How to apply:** on any flag:money item, check every cents assert uses isSafeInteger, guard sums after the reduce and products before division (magnitude check when fractional), and require at least one probe that actually crosses 2^53; when a fixer deviates from a prescribed money-path fix with reasoning, re-gate with a scoped re-review (parallel with scoped QA) rather than accepting or full-looping.
- 2026-07-22 A clean `tsc` + `next build` does NOT prove client-side hooks exist at runtime — an engineer used React 19's `useActionState` on a React 18 repo; types/build passed, both editor routes 500'd in production, and only QA's live smoke caught it — **Why:** bundlers/types can resolve an API that the installed runtime lacks, and unit tests rarely execute client hook wiring. **How to apply:** builders must copy an existing working form/component's hook pattern from the SAME repo before writing a new one; QA's live smoke must hit every NEW route (not just changed ones) even when the suite is green. Also: any UI-polish agent given a "token-only colors" rule needs an executable grep for raw hex in its re-gate — the instruction alone was violated the same run it was given.
- 2026-07-22 A Next.js public page that displays admin-toggled DB content (published banners, feature flags, announcements) will silently never update if it is statically prerendered — build, typecheck, and unit tests all pass while the page is baked at build time with the toggle off — **Why:** App Router defaults pages with no dynamic APIs to static; the DB query runs once at build, so request-time renders never re-query, and only a live smoke (publish → GET page) catches it. **How to apply:** when an item wires DB-driven content into a previously-static page, the builder must set the rendering mode explicitly (`force-dynamic`, `revalidate`, or revalidatePath-on-mutate) and QA's live smoke must assert the toggle round-trip against the built (`next build && next start`) server, not the dev server.
- 2026-07-22 Any dedupe/identity key built from an email address must canonicalize (`lower(trim())`) in BOTH the app-side key builder and the SQL grouping, via one shared helper — and "remind N days after last event" schedulers should derive eligibility from the DB at fire time (cron scan on MAX(event_at)) rather than cancel/reschedule queued jobs — **Why:** raw `contact_email` split `Bob@x.com`/`bob@x.com` into two identities, defeating a PK-based send ledger (double-send) — caught only by Opus review, not QA; and pg-boss `singletonKey` REJECTS duplicate enqueues rather than retiming the pending job, so event-scheduled supersede designs silently fail the "newer event pushes the reminder out" invariant. **How to apply:** on any flag:data-path item keying on user-supplied strings, review every key builder AND every SQL GROUP/DISTINCT site for canonicalization drift (test: mixed-case variant collapses to one row); prefer scan-derived eligibility for last-event-relative schedules; verify a queue library's singleton/replace semantics from its types/docs before betting a correctness invariant on them.
- 2026-07-22 A QA test asserting a NEGATIVE side effect ("no ledger row is written when X") must drive the real production entry point, not re-derive the property from adapter calls — if the function under test is unexported, export a test seam (or drive it through the queue/route) rather than testing around it. **Why:** QA "confirmed" skip-without-claim by checking the adapter returned null and the ledger was empty, without ever invoking the handler; the handler actually claimed the key before the check (its comment lied), and only review caught it. A passing absence-assertion that never executes the guarded path validates the spec, not the code. **How to apply:** in QA spawn prompts for any ordering/absence criterion, require the test to call the actual handler/action and name the seam; reviewers should cross-check code comments claiming ordering guarantees against the literal statement order.
- 2026-07-22 When a plan item requires touching live hosted infra (applying a migration, hitting a DB port), the orchestrator should probe the environment's actual capability itself BEFORE gating — try the ports/tokens, then state the adjudicated constraint in QA's spawn prompt ("X is impossible from this machine; the correct handling is loud skip-with-note + auto-activate flag + accurate PROGRESS deferral; overclaiming = FAIL") — **Why:** an inherited live-smoke item had an un-appliable migration (all PG ports egress-blocked, no management token); without up-front adjudication QA would have FAILed on an environment fact no builder can fix, burning attempts toward a false BLOCKED. **How to apply:** 2 minutes of nc/env probes by the orchestrator converts an unfixable criterion into a verified-degraded gate; require the suite to probe the capability at runtime and auto-activate the assertion when provisioning lands, and require PROGRESS to keep that check DEFERRED rather than claiming live verification.
